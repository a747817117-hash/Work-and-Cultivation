> 分析时间：2026-05-30 17:36

---

## 一、需求意图

### 1.1 需求目标

创建一个"挂机修仙"网页游戏，上班族可以登录进入房间，与其他玩家一起挂机修仙，系统定时根据房间人数生成故事推动角色成长，最终阶层最高者获胜。

### 1.2 核心功能点

1. **用户系统**：注册/登录/个人账号管理
2. **房间系统**：创建房间、加入房间、房间内玩家列表
3. **角色属性系统**：能力、习惯、偏好、兴趣、武器、性格（每人不同）
4. **故事生成引擎**：每N分钟（可配置）根据房间人数生成故事
5. **阶层晋升系统**：修仙境界/阶层，故事推动角色变化和晋升
6. **挂机玩法**：玩家只需在线观看，自动推进
7. **胜利判定**：结束条件可配置（故事发生数），阶层最高者获胜

### 1.3 业务背景

面向上班族的休闲挂机游戏，同事之间可以在工作间隙一起"修仙"，增加办公室趣味性。

## 二、关联模块分析

### 2.1 涉及模块

| 模块 | 说明 | 核心功能 |
|------|------|---------|
| **用户模块** | 注册/登录/会话管理 | JWT认证、用户信息CRUD |
| **房间模块** | 房间创建/加入/退出 | 房间列表、房间内玩家管理 |
| **角色模块** | 角色属性/状态管理 | 属性生成、阶层晋升、状态存储 |
| **游戏引擎** | 定时任务/故事生成 | 定时器、AI调用、故事推送 |
| **前端UI** | 页面/交互/实时展示 | 登录页、房间页、游戏页 |

### 2.2 数据表设计

| 表名 | 字段 | 说明 |
|------|------|------|
| **users** | id, username, password_hash, created_at | 用户表 |
| **rooms** | id, name, creator_id, max_stories, status, created_at | 房间表 |
| **characters** | id, user_id, room_id, name, ability, habit, preference, interest, weapon, personality, level, exp | 角色表 |
| **stories** | id, room_id, sequence_num, content, created_at | 故事记录表 |
| **game_events** | id, room_id, character_id, story_id, event_type, description, level_change | 游戏事件表 |

### 2.3 API接口设计

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/auth/register | POST | 用户注册 |
| /api/auth/login | POST | 用户登录 |
| /api/rooms | GET/POST | 房间列表/创建房间 |
| /api/rooms/:id | GET | 房间详情 |
| /api/rooms/:id/join | POST | 加入房间 |
| /api/rooms/:id/start | POST | 开始游戏 |
| /api/rooms/:id/status | GET | 房间游戏状态 |
| /ws/rooms/:id | WebSocket | 实时故事推送 |

## 三、风险点识别

| 风险类型 | 风险描述 | 严重程度 | 建议措施 |
|---------|---------|---------|---------|
| **AI API依赖** | 故事生成依赖外部AI服务，API不可用会导致游戏中断 | 中 | 设置超时重试、降级到预设模板 |
| **并发写入** | 多人同时加入房间/触发事件可能产生竞态条件 | 中 | 使用数据库事务、乐观锁 |
| **WebSocket稳定性** | 长连接可能因网络问题断开 | 低 | 实现自动重连机制 |
| **AI成本** | 频繁调用AI API可能产生较高费用 | 中 | 设置调用频率限制、使用缓存 |
| **内存泄漏** | 定时器未清理可能导致内存泄漏 | 中 | 确保房间关闭时清理所有定时器 |
| **SQLite并发** | SQLite并发写入能力有限 | 低 | 使用WAL模式、控制写入频率 |

## 四、边界条件

| 场景 | 异常情况 | 建议处理方式 |
|------|---------|-------------|
| **空数据** | 房间内无人/只有1人 | 至少2人才能开始游戏，提示人数不足 |
| **玩家中途退出** | 游戏进行中玩家离开房间 | 角色保留但标记"离线"，继续参与故事 |
| **重复加入** | 同一用户重复加入同一房间 | 返回"已在房间内"提示 |
| **AI生成失败** | AI API调用超时或返回错误 | 重试3次后使用预设模板兜底 |
| **极端值** | 房间人数超过上限 | 设置房间最大人数限制（如10人） |
| **故事数上限** | 达到配置的故事发生数 | 自动结束游戏，宣布获胜者 |
| **同时开始** | 多人同时点击开始 | 使用互斥锁，只有房主能开始 |
| **长时间无操作** | 玩家挂机但网络断开 | 心跳检测，超时标记离线 |

| 筑基期 | 筑就修道根基，寿命延长，可御物飞行 |
| 金丹期 | 凝结金丹，寿元大增（通常五百年以上），实力质变 |
| 元婴期 | 金丹化元婴，可离体短暂存活，神通广大 |
| 化神期 | 领悟天地法则，元神强大，开始接触天道 |
| 炼虚期 | 破碎虚空，炼化法则为己用 |
| 合体期 | 元神与肉身完美合一，掌控一方天地 |
| 大乘期 | 准备渡劫，法力通玄，近乎仙人 |
| 渡劫期 | 经历天劫考验，渡过则飞升成仙 |
| 真仙 | 飞升后的仙人境界 |

### 5.2 角色属性生成（已确认）

- 玩家输入：角色名字
- 系统随机生成：能力、习惯、偏好、兴趣、武器、性格

### 5.3 故事风格（已确认）

- 混合风格：搞笑与正经并存，根据场景自适应

### 5.4 AI服务（已确认）

- 使用OpenAI兼容接口
- 可通过环境变量配置API地址和Key

### 5.5 部署方式（已确认）

- Node.js + Express + SQLite
- 前端原生HTML/CSS/JS

## 六、技术架构

### 6.1 推荐技术栈

| 层级 | 技术选择 | 说明 |
|------|---------|------|
| **前端** | HTML/CSS/JS + WebSocket | 轻量级，无需框架 |
